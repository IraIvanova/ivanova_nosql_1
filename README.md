# Відповіді на теоретичні питання

## Частина 1 — Завантаження даних та проєктування схеми
### 1. Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?

Аудіо-характеристики логічно належать до однієї групи даних, тому їх зручно зберігати окремо у вкладеному об’єкті `audio_features`.

Таке вкладення вигідне з погляду логічності структури: усі пов’язані параметри зібрані разом, і до них можна звертатися через dot notation, наприклад `audio_features.tempo`.

Проблеми можуть виникнути, якщо вкладений об’єкт стає занадто великим або якщо потрібно часто оновлювати чи індексувати багато вкладених полів окремо.

### 2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?

Масив дозволяє працювати не з одним рядком, а з кожним артистом окремо.

Таким чином стає простіше знаходити всі треки конкретного виконавця, групувати треки та виконувати фільтрацію за кожним окремим артистом.
Це значно зручніше, ніж працювати з одним рядком, який містить усіх артистів одразу.

### 3. Що таке `$out` і чим він відрізняється від `$merge`? Коли використовувати кожен?
`$out` записує результат агрегації в окрему колекцію та фактично створює її заново. Якщо колекція вже існує, її дані будуть замінені новими.

`$merge` дозволяє не перезаписувати всю колекцію, а додавати нові документи або оновлювати вже існуючі.

## Частина 2 — Запити до даних
### 1. Для чого використовується інструкція `$unwind`?

`$unwind` використовується для розгортання масиву в окремі документи.

У нашому випадку, коли трек має декількох артистів у масиві `artists`, після застосування `$unwind` кожен артист буде оброблятися окремо.

Відповідно, можна групувати, рахувати або фільтрувати дані за елементами масиву.

### 2. Чим `$stdDevPop` відрізняється від `$stdDevSamp`?

`$stdDevPop` рахує стандартне відхилення для всієї сукупності даних. Його варто використовувати, коли ми маємо всі потрібні дані.

`$stdDevSamp` рахує стандартне відхилення для вибірки. Його використовують, коли дані є лише частиною сукупності.

## Частина 3 — Аналітика через Aggregation Pipeline
### 1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.

Якщо знизити поріг до 1 треку:
- отримаємо список виконавців, які мають лише один трек з високою популярністю;
- рейтинг буде менш надійним, оскільки базуватиметься лише на одному треку.

Якщо вибирати виконавців із більш ніж 50 треками:
- отримаємо більш надійний рейтинг, оскільки він базуватиметься на більшій кількості треків;
- але буде відсіяна частина виконавців, у яких загалом немає такої кількості треків, особливо нових або менш популярних артистів.

Якщо знизити поріг до 50 треків:

## Частина 4 — Індекси та оптимізація
### 1. Що змінилося в плані виконання?
### 2. Як зрозуміти, що індекс використовується?

До створення індексу MongoDB перевіряла всю колекцію, щоб знайти документи, які відповідають умовам запиту.

```
{
  winningStage: 'SORT',
  inputStage: 'COLLSCAN',
  totalKeysExamined: 0,
  totalDocsExamined: 113999,
  executionTimeMillis: 136
}
```
Це видно за етапом `COLLSCAN`. Також для сортування за `popularity` виконється окрема операція сортування.

Після створення складеного індексу MongoDB почала використовувати індекс для пошуку документів. 
```
{
  winningStage: 'FETCH',
  inputStage: 'IXSCAN',
  totalKeysExamined: 412,
  totalDocsExamined: 354,
  executionTimeMillis: 2
}
```
Замість `COLLSCAN` з’явився етап `IXSCAN`, тобто сканування індексу. Кількість переглянутих документів зменшилася з 113999 до 354, а час виконання запиту - з 136 мс до 2 мс.
Зрозуміти, що індекс використовується, можна за значенням totalKeysExamined, яке показує кількість переглянутих ключів індексу.
Після створення індексу в explain() значення totalKeysExamined стало більшим за 0, що підтверджує використання індексу MongoDB.

### Завдання 3. Чи є цей запит покривним (covered query)? Надайте розгорнуту та обґрунтовану відповідь у файлі README.
Дано запит:
```
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
});
```

Індекс із завдання 1 має таку структуру:

```
{
  track_genre: 1,
  popularity: -1,
  "audio_features.danceability": 1
}
```

У цьому запиті використовуються поля `track_genre` і `popularity`, які входять до індексу. Але сам запит **не є покривним**, тому що він повертає повні документи з колекції `tracks`.

Для covered query недостатньо, щоб поля з умови були в індексі. Якщо не вказати проєкцію, MongoDB повертає весь документ. Цих полів немає повністю в індексі, тому MongoDB має звертатися до самих документів.

```
{
  winningStage: 'FETCH',
  inputStage: 'IXSCAN',
  totalKeysExamined: 317,
  totalDocsExamined: 317,
  executionTimeMillis: 3
}

```

У `explain()` це можна побачити за етапом `FETCH`. Якщо є `FETCH`, це означає, що MongoDB після сканування індексу додатково читає документи з колекції.

Щоб запит став покривним, потрібно додати проєкцію тільки на поля, які є в індексі, і прибрати `_id`:

```js
db.tracks.find(
  {
    track_genre: "pop",
    popularity: { $gte: 70 }
  },
  {
    _id: 0,
    track_genre: 1,
    popularity: 1
  }
);
```

У такому випадку MongoDB може отримати всі потрібні дані безпосередньо з індексу, без читання повних документів із колекції.

```
{
  winningStage: 'PROJECTION_COVERED',
  inputStage: 'IXSCAN',
  totalKeysExamined: 317,
  totalDocsExamined: 0,
  executionTimeMillis: 0
}
```

В плані виконання з’явився етап `PROJECTION_COVERED`. Це означає, що MongoDB змогла отримати всі необхідні поля безпосередньо з індексу, без звернення до документів колекції.
Також це підтверджує `totalDocsExamined: 0` - MongoDB не читала документи з колекції, а отримала результат тільки з індексу.
